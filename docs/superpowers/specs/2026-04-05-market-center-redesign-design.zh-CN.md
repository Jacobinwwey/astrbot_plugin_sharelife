# 2026-04-06 v1.2 - Sharelife 市场中心入口重构设计

## 时间与版本

- 日期：2026-04-06
- 版本：v1.2
- 状态：已按实现基线更新
- 当前语言：简体中文

  - 英文主文件：`docs/superpowers/specs/2026-04-05-market-center-redesign-design.md`

### 背景

`/market` 已经是 Sharelife WebUI 的独立路由，但当前页面仍然把两种不同职责混在一起：

1. 公开市场目录浏览。
2. 成员侧的试用、安装、提交以及本地状态相关操作。

这会同时削弱两个方向的体验。它既不像一个清晰的公开市场入口，也让只想浏览资源的用户在首屏承受了过多操作负担。

用户已经为这次市场中心深度重构明确了以下产品约束：

1. 在全新 clone 和干净分支上开展工作。
2. 以 `/market` 作为本次改造的主切片。
3. 首屏首先必须是公开只读入口。
4. 成员动作必须后移到点击卡片后的细则层。
5. 第一阶段不做常驻对比能力。
6. 参考高密度控制台方向，但要重排信息层级。
7. 后续通过 Stitch MCP 至少生成五个细则层方案，调用后等待五分钟再获取，并且这些方案之间要能一键切换对比。

### 目标

把 `/market` 改造成一个以浏览和发现为优先的公开市场入口，同时在用户打开具体 pack 后，仍然能够顺畅衔接到鉴权后的成员动作。

### 范围

本设计涵盖：

1. 独立 `/market` 页面的信息架构。
2. 公开目录卡片层的 contract。
3. 点击卡片后承接成员动作的细则层 contract。
4. 手工实现的市场入口与后续 Stitch 细则层变体之间的分阶段边界。
5. 为了保留现有 runtime ID、API 绑定和测试而必须遵守的关键约束。

本设计不涵盖：

1. 同一切片内对 `/`、`/member`、`/admin` 的联动重构。
2. 第一阶段重新加入常驻对比。
3. 除非现有 payload 真的阻碍 UI contract，否则不主动改造后端市场 API。
4. 在 Stitch 生成前就手写最终版细则层视觉。
5. 永久性的排序系统、个性化推荐或收藏系统。

### 备选方案

#### 方案 A：编辑策展型公开入口

优点：

1. 最像公开落地页。
2. 更容易高位展示精选 pack 和可信背书信息。

缺点：

1. 不利于高密度目录浏览。
2. 与当前市场页结构差距过大。
3. 第一阶段的布局改动量偏大。

#### 方案 B：策展目录控制台

优点：

1. 最贴合参考方向，也最契合当前 `/market` 代码形态。
2. 在保留高密度浏览效率的同时，能清晰收紧动作边界。
3. 后续 Stitch 只改细则层时，列表层可以保持稳定。

缺点：

1. 必须仔细处理视觉层级，否则仍可能看起来像本地操作台。

#### 方案 C：品牌落地页与结果区混合

优点：

1. 品牌表达比纯控制台更强。
2. 更容易强调审核流程与可信度说明。

缺点：

1. 首屏更重。
2. 可能削弱搜索与目录浏览效率。
3. 第一阶段的重构成本更高。

### 决策

采用方案 B：策展目录控制台。

独立市场页保留高效的目录控制台结构，但页面语义要从“混合操作仪表板”切换为“公开市场入口”。公开浏览是首屏主任务，成员动作后移到点击卡片后的细则层。

### 设计

#### 1. 入口定位

`/market` 要成为 Sharelife 市场的公开只读前门。

首屏进入时，页面应突出：

1. Sharelife Market 是什么。
2. 应该如何浏览它。
3. 为什么这个目录是可信的。

首屏不应突出：

1. 本地安装状态刷新。
2. 立即执行安装或上传的动作。
3. 在用户尚未选中具体 pack 之前就强调角色操作。

#### 2. 信息架构

首屏结果区上方应重组为三层结构：

1. 品牌与上下文头部。
2. Spotlight 风格的全局搜索。
3. 结果控制行。

其下保留两栏式主体结构：

1. 左侧紧凑筛选栏。
2. 右侧卡片网格。

#### 3. 头部层

头部层应比当前 top utility bar 更精简，也更偏公开站语义。

它应包含：

1. `Sharelife` 品牌标识与市场标题。
2. 一句简短的公开市场定位文案。
3. locale 切换。
4. 文档入口以及其他公开只读链接。

它应从首屏移除或明显降权：

1. `Refresh Local Installations`。
2. 成员或管理员操作强调。
3. 上传、安装、试用类 CTA。

如果保留跨控制台跳转链接，它们也应该是次级导航，而不是主动作。

#### 4. Spotlight 风格搜索层

搜索框应成为页面最强的视觉锚点。

位置要求：

1. 紧贴主入口头部下方。
2. 位于排序/结果控制行上方。
3. 宽度足够大，视觉气质更接近 Spotlight 或 Raycast，而不是普通表格过滤输入框。

行为要求：

1. 嵌入页面，不做浮层式全局弹窗。
2. 搜索范围覆盖 `pack_id`、类似标题字段、maintainer、tags、review labels、warning flags、compatibility，以及可用的摘要文本。
3. 只更新当前网格结果，不把页面导向一个鉴权优先的操作流程。

产品意图：

1. 让“发现”成为首屏主感知。
2. 降低当前页面“首先像操作控制台”的感觉。

#### 5. 结果控制层

搜索框下方这一行只承载结果控制能力。

保留：

1. 排序。
2. 结果数。
3. 如果当前代码已有干净支持，可保留视图切换。

不要混入这一行的内容：

1. 安装动作。
2. 上传动作。
3. 鉴权优先控制。
4. 本地安装刷新。

如果仍然需要刷新能力，也应重命名为目录刷新，而不是本地安装刷新。

#### 6. 筛选栏

左侧筛选栏应保留，因为它既契合选定的控制台方向，也贴合现有页面结构。

它应限制在公开浏览相关维度，例如：

1. Pack type。
2. Risk level。
3. Compatibility。
4. Featured 状态。
5. Review labels。
6. Warning flags。

它不应演化成第二块成员操作控制台。

#### 7. 目录卡片 Contract

公开卡片层只回答一个窄问题：

这是什么 pack，风险如何，是否值得点开看更多细节。

每张卡片只暴露浏览所必需的公开只读元数据：

1. Pack 标识或显示标题。
2. Pack type。
3. 版本。
4. Maintainer。
5. Compatibility 状态。
6. 风险等级。
7. 简短摘要。
8. 少量 tags 与 review/warning 指示。
9. 合适时可显示轻量指标，例如 section 数、label 数、更新时间。

卡片层不应直接出现以下成员动作按钮：

1. Try。
2. Install。
3. Upload。
4. Member login 或本地环境检查。

卡片层的主动作应收敛为单一细则入口，例如：

1. 点击整个卡片。
2. 点击 `View Details` 按钮。

#### 8. 细则层 Contract

点击卡片后，用户应进入第二层表面。

默认形态：

1. 桌面端使用右侧抽屉。
2. 移动端使用全屏 sheet。

这一层才是更深信息与成员动作的承载面。

它应成为以下内容首次出现的位置：

1. Try。
2. Install。
3. Upload 相关动作或入口。
4. 成员态环境提示。
5. 必要时的鉴权提示。
6. 真正执行安装前的 section 同步选择能力。

公开细节内容在未登录状态下仍应可读。只有当用户触发 Try、Install、Upload 等成员动作时，才要求鉴权。

Install 在细则层中必须支持按 section 选择同步范围。`memory_store`、`conversation_history`、`knowledge_base` 这类有状态或本地数据 section 应被明确表达为可选同步目标，用户可以在安装时主动跳过。

细则层应消费一个稳定的数据 contract，并尽量沿用现有 catalog/detail payload 形状：

1. `pack_id`
2. 如有则使用 display title
3. `version`
4. `maintainer`
5. `pack_type`
6. `risk_level`
7. `compatibility`
8. `review_labels`
9. `warning_flags`
10. `sections`
11. 摘要或描述
12. 更新时间类元数据
13. API 已返回的公开 audit 或来源事实

即使后续通过 Stitch 更换视觉布局，这个细则层 contract 也必须保持稳定。

#### 9. 两阶段交付

##### 阶段 1：手工实现公开市场入口重构

实现内容：

1. 头部重排。
2. Spotlight 风格搜索的位置调整。
3. 干净的结果控制行。
4. 公开筛选栏。
5. 公开卡片 contract。
6. 点击卡片后打开的细则层骨架。

本阶段不实现：

1. 常驻对比。
2. 高完成度的细则层最终视觉方案。
3. 其他 WebUI 页面的大范围联动重构。

##### 阶段 2：通过 Stitch 生成细则层变体

当主入口和细则层 contract 稳定后，再只针对细则层使用 Stitch MCP。

首屏市场入口仍然由生产环境的 HTML/CSS/JS 手工集成。被探索变体的视觉面是细则层，而不是整个市场首页。

#### 10. Stitch 生成规则

当项目进入细则层生成阶段时，遵循以下规则：

1. 单次目标至少生成五个变体。
2. 这五个变体必须服务同一个细则层任务，而不是五个彼此无关的页面。
3. 调用 Stitch 后，等待五分钟再抓取结果。
4. 如果 Stitch 返回的是建议式输出而非最终变体，可以接受必要的后续路径，但目标仍然是拿到至少五个可用方案。
5. 这些方案必须能在同一个已打开 pack 的上下文里方便对比。
6. 生成出来的细则层方向必须预留安装时按 section 同步的表达空间，不能暗示所有已声明 section 都是强制同步项。

生产要求不是“五个互相割裂的静态稿”，而是“同一个细则层有五种可以一键切换比较的方案”。

建议的生产交互：

1. 使用分段控制条或 tab strip，例如 `Variant 1`、`Variant 2`、`Variant 3`、`Variant 4`、`Variant 5`。
2. 切换时保持相同 pack 上下文，只替换细则层表现形式。

##### Stitch MCP 调用前置门槛

在这个切片里，任何 Stitch MCP 调用发生前，都必须先满足以下前置条件：

1. 团队先把本次目标对应的 Stitch MCP 完善指令明确下来。
2. 这份指令不能只停留在聊天记录里，必须落成模块内的 `Design.md`。
3. `Design.md` 必须放在拥有该 Stitch 生成界面的前端调整模块独立文件夹下。
4. 对于这次市场中心切片，Stitch 接管的目标是细则层，而不是首屏市场入口。
5. 这份模块内 `Design.md` 至少要写清：
   1. 目标界面与用户任务
   2. 必须遵守的数据 contract
   3. 不能破坏的 DOM / runtime 约束
   4. 最终确认后的 Stitch MCP prompt 或指令块
   5. 期望生成的变体数量与对比方式
   6. 生成后的验收检查项
   7. 最新确认的市场中心基线，以及当前安装时按 section 同步的规则
6. 只有在该 `Design.md` 创建并评审完成后，才允许真正发起 Stitch 生成调用。

#### 11. 需要保留的运行时约束

重构应尽量保留当前架构边界：

1. 继续使用现有独立 `/market` 路由。
2. 除非必须做聚焦拆分，否则优先复用 `market.html`、`market_page.js`、`market_cards.js` 与 `style.css` 的现有结构。
3. 不应随意破坏当前 JS 和测试依赖的 DOM ID 与钩子；若必须改名，必须有测试支撑。
4. 保持现有 WebUI i18n 行为一致。
5. 保持运行时目录数据与 public snapshot 数据之间的现有来源策略。

#### 12. 风险

主要风险：

1. 如果旧动作仍然视觉占优，页面仍可能显得过于操作化。
2. 如果搜索只过滤 pack ID 而忽略其他公开元数据，搜索深度会不足。
3. 如果细则层过度耦合到临时手写布局，后续 Stitch 替换成本会偏高。
4. 如果改动移除了必要的 ID 或容器，现有 JS 绑定可能失效。

缓解方式：

1. 把高优先级成员动作从首屏移除。
2. 预先明确并测试搜索字段。
3. 把细则层当作“稳定数据 contract + 可替换表现层”处理。
4. 先补有针对性的 WebUI 测试，再改生产代码。

#### 13. 测试策略

在改生产代码前，先增加或扩展以下测试：

1. 支撑新公开卡片 contract 的 market card view-model 行为。
2. 基于选定元数据字段的搜索行为。
3. 首屏公开浏览与细则层成员动作之间的边界。
4. 在 Stitch 视觉落地前，新引入的细则层状态与变体切换骨架。

视觉验证应确认：

1. 搜索框是最主要的视觉锚点。
2. 首屏读起来是公开目录，而不是成员操作页。
3. 卡片上没有直接暴露 install、trial、upload 动作。
4. 点击卡片是进入更深动作层的清晰路径。

### 成功标准

当满足以下条件时，这个重构切片算完成：

1. `/market` 首屏读起来是公开只读市场入口。
2. Spotlight 风格的全局搜索位于入口头部下方、排序行上方。
3. 首屏支持浏览、筛选和搜索，但不在卡片上直接暴露成员动作。
4. 点击卡片后会打开一个稳定的细则层骨架，用于承接成员动作。
5. 细则层 contract 稳定到足以支持后续 Stitch 生成，而无需重写列表层。
6. 后续 Stitch 步骤能够生成至少五个细则层变体，并支持一键切换对比。
