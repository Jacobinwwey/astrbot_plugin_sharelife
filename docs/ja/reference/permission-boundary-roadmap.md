# 権限制御境界とロール分離ロードマップ（User / Reviewer / Admin / Developer）

> 基準バージョン：`v0.3.x`  
> 最終固定日：`2026-04-04`  
> 対象：Sharelife WebUI、API、コマンド面、VitePress ドキュメント構成

## 1. 目的と原則

このロードマップは、Sharelife を「機能積み上げ型コンソール」から「ロール境界が明確で検証可能な認可モデル」へ移行するための実行基準です。

基本原則：
- 機能追加より先にロール分離：誰が実行できるかを先に確定する。
- ドキュメントは認可契約：文書上の権限表示はコードのガード条件と一致させる。
- 最小権限を既定値にする：各ロールに必要最小限だけを公開する。

## 2. ロール権限マトリクス（固定版）

| ロール | 許可 | 禁止 |
| --- | --- | --- |
| User | マーケット検索、インストール/削除、アップロード/ダウンロード、進捗閲覧 | 審査決裁、strict apply、rollback、レビュアー端末キー管理 |
| Reviewer | 審査キュー処理、リスクラベル付与、審査決裁、端末キー自己管理（最大3台） | システム級 apply/rollback、グローバル設定変更 |
| Admin | ライブラリ全体管理、公開運用、破壊的操作、レビュアー昇格キー配布 | なし（監査対象） |
| Developer | 設計拡張、API 拡張、テスト/文書保守 | 実行時の認可バイパス |

補足制約：
- Reviewer 端末上限：既定 `3`（設定で変更可能）。
- `reviewer_id` ごとに同時有効セッションは 1 つ。
- 端末キーを失効すると関連トークンは即時失効する。

## 3. キー配布と更新戦略

### 3.1 配布フロー
1. Admin が招待キーを発行（短 TTL、使い切り）。
2. Reviewer がキーを交換して `reviewer_id` を紐付け。
3. Reviewer が信頼端末キーを登録。
4. ログイン時に `reviewer_id + reviewer_device_key + reviewer_password` を検証。

### 3.2 ローテーションと失効
- 招待キー：短寿命 + 単回使用。
- Reviewer セッショントークン：既定 TTL 7 日。
- 端末キー：削除・置換可能。上限超過時は先に削除が必要。
- インシデント時：Admin が招待キーとセッションを強制失効可能。

## 4. フロントエンド / バックエンド / 文書の作業面

### 4.1 フロントエンド（WebUI）
- 完了済み：
  - Reviewer 専用ページとロール別導線を実装。
  - member / reviewer / admin の表示境界を明確化。
- 次工程：
  - 破壊的操作は admin コンテキストのみに限定。
  - Reviewer には証跡を維持しつつ、既定表示の技術ノイズを削減。

### 4.2 バックエンド（API / WebUI server）
- 完了済み：
  - `reviewer` 認証と端末キー register/list/revoke API を実装。
  - 互換性のため `user -> member` マッピングを維持。
  - 未許可ロールの制限ルートアクセスを `403` で統一。
  - `GET /api/admin/audit` に reviewer/device/action 単位の監査集計サマリーを追加。
- 次工程：
  - 高リスク API にレート制限と異常検知を追加。

### 4.3 ドキュメント（VitePress）
- 完了済み：
  - User / Reviewer / Administration / Developers の4区分でサイドバー再編。
  - Reviewer 端末キー SOP を `zh/en/ja` で公開。
- 次工程：
  - API ページに Auth Badge（例：`System.Admin`）を必須化。
  - 破壊的操作ページに警告とロールバック手順を明記。

## 5. 受け入れ基準

以下をすべて満たした場合のみ、この段階を完了とする：
1. reviewer 端末/セッションを含むインターフェース試験が通過する。  
2. 非許可ロールは制限操作で安定して `403` を返す。  
3. 3 言語すべてでロールドメイン入口が揃っている。  
4. 追加 API に必要ロールと拒否挙動が文書化されている。  

## 6. 次段階の重点

1. LDAP/OAuth 連携を追加しつつ、ローカル Reviewer Key をフォールバック維持。  
2. リスク級別ベースの審査キュー自動振り分け。  
3. 権限変更監査の可視化（担当者、端末、操作）。  
4. 文書とコードの CI 連動（Auth Badge 欠落を失敗扱い）。
