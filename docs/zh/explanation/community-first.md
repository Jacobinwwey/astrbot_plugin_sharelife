# 为什么先做社区场景

`sharelife` 的 v1 先把个人用户和小社区场景跑稳，企业级排班和复杂审批先不放进默认流程。

## 取舍

1. 先交付试用、风险扫描、管理员 dry-run/apply/rollback 这条主链路。
2. 企业化治理能力先保留扩展接口，不作为默认工作流。
3. 提前保持模块边界清晰，降低后续迁移到 SDK v4 的成本。

## 当前收益

1. 普通用户可以先试用模板，不直接改全局运行态。
2. 管理员有明确的高风险变更入口，操作可追踪。
3. 社区模板可以在严格模式下被引入，而不是“直接覆盖配置”。

## 为什么后续可扩展

1. 现有治理服务与适配层可复用到企业模式。
2. 中英日三语从一开始并行，后续扩语种不需要重做结构。
