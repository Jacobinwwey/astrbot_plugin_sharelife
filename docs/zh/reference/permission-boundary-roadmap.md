# 权限边界与职责解耦路线图（用户 / 审核员 / 管理员 / 开发者）

> 版本基线：`v0.3.x`  
> 最近固化：`2026-04-04`  
> 适用范围：Sharelife WebUI、API、命令面、VitePress 文档体系

## 1. 目标与原则

本路线图用于把 Sharelife 从“功能堆叠型控制台”推进到“角色边界清晰、权限可验证、文档可执行”的治理结构。

核心原则：
- 角色隔离优先于功能堆叠：先定义谁能做，再定义怎么做。
- 文档是权限契约：文档中的权限标签必须能映射到代码中的鉴权中间件。
- 默认最小权限：普通用户只暴露必要能力，高危入口仅对审核与管理员可见。

## 2. 角色权限矩阵（冻结版）

| 角色 | 允许操作 | 禁止操作 |
| --- | --- | --- |
| 普通用户（User） | 市场检索、安装/卸载、上传/下载、进度浏览 | 审核决策、严格应用、回滚、设备密钥管理 |
| 审核员（Reviewer） | 审核队列处理、风险标注、审核结论提交、设备密钥自管理（最多 3 台） | 系统级 apply/rollback、全局配置写入 |
| 管理员（Admin） | 全量库管理、发布治理、危险操作执行、审核员准入分发 | 无（受审计约束） |
| 开发者（Developer） | 架构演进、接口扩展、测试与文档维护 | 运行时绕过鉴权 |

补充约束：
- 审核员设备上限：默认 `3` 台，可配置。
- 审核员同一时刻仅允许一个活跃会话（Single Active Session）。
- 撤销设备时，关联会话令牌立即失效。

## 3. 密钥与分发策略（实施版）

### 3.1 分发链路
1. 管理员生成邀请码（短时效、一次性使用）。  
2. 审核员兑换邀请码后绑定 `reviewer_id`。  
3. 审核员在可信设备注册设备密钥。  
4. 登录时校验：`reviewer_id + reviewer_device_key + reviewer_password`。

### 3.2 刷新与轮换
- 邀请码：短 TTL + 一次性消费。
- 审核员会话 Token：默认 7 天，到期必须重新登录。
- 设备密钥：支持手动撤销和替换；超过上限必须先删除旧设备。
- 异常策略：发现泄露时，管理员可强制失效邀请码和审核员会话。

## 4. 前后端结构改造（当前完成 + 后续）

### 4.1 前端（WebUI）
- 已完成：
  - 独立 Reviewer 页面与角色入口隔离。
  - `user/member` 与 `admin/reviewer` 入口边界明确。
- 后续：
  - 所有高危操作入口强制收纳到管理员上下文。
  - 审核工作区保留“风险证据”但默认隐藏底层技术噪音。

### 4.2 后端（API / Server）
- 已完成：
  - `reviewer` 角色鉴权路径、设备注册/列出/撤销流程。
  - `user -> member` 兼容映射，避免旧入口中断。
  - 非授权角色访问管理路径时统一 `403`。
  - `GET /api/admin/audit` 已支持按 reviewer_id / device_id / action 的聚合审计摘要。
- 后续：
  - 为关键接口增加更细粒度速率限制与异常告警。

### 4.3 文档（VitePress）
- 已完成：
  - 侧边栏按 User / Reviewer / Administration / Developers 四域分层。
  - 审核员设备密钥文档三语同步。
- 后续：
  - API 参考页强制附加 Auth Badge（例如 `System.Admin`）。
  - 每个高危操作加入前置风险声明与回滚指引。

## 5. 验收标准（对齐 CI）

满足以下条件才可视为“权限边界阶段完成”：
1. 接口与 WebUI 测试全部通过（含 reviewer 设备链路）。  
2. 未授权角色访问管理路径稳定返回 `403`。  
3. 三语文档存在同名角色分层入口，不再混合描述。  
4. 新增接口在 API 文档中标注角色权限与拒绝语义。  

## 6. 后续迭代方向（下一阶段）

1. 引入可选外部身份源（LDAP/OAuth）与本地 Reviewer Key 兼容模式。  
2. 审核策略引擎化（按风险级别自动分配审核队列）。  
3. 权限变更审计可视化（时间线 + 责任人 + 设备）。  
4. 文档与代码联动校验：缺失 Auth Badge 的 API 页面在 CI 直接失败。
